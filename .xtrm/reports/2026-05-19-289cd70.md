## Iteration 1 — vocabulary-only skill draft

Built initial SKILL.md with 6-step workflow (scan → map → detect → propose → approve → apply), 8 detection patterns (A-H), relationship vocabulary cheat-sheet, escalation to overthinker for >50-issue boards.

Eval harness: `bash seed-board-a.sh /tmp/triage-evals/<dir>` creates a 16-issue fixture with deliberately overlapping bug+feature pairs, near-duplicates, 3-sibling cluster, test↔impl pair, spawned follow-up, priority mismatch, in-progress claim, control issue, and a closed near-dup. Each subagent gets its own fixture dir + auto-picked Dolt port.

Two subagents (`general-purpose`) ran in parallel: with-skill vs no-skill baseline.

| Metric | with_skill | baseline (no skill) | Δ |
|--------|------------|---------------------|---|
| Pass rate | 10/10 | 9/10 | +10pp |
| Tokens | 57,457 | 50,344 | +14% |
| Duration | 124.3 s | 103.8 s | +20% |

The single failed baseline assertion: it didn't use `--type=validates` for the test↔impl pair (defaulted to plain `blocks`). Honest read: a smart LLM does ~90% of triage without any skill. The skill was a vocabulary nudger more than a workflow shaper. Filed back to the user → decision to widen value-add.

## Iteration 2 — four affordances the baseline can't reach

Added to SKILL.md:
1. **Step 2b GitNexus reinforcement.** Mandatory `gitnexus_query`/`gitnexus_context` attempt for clusters sharing surface tokens; explicit "no GitNexus reinforcement" fallback flag if index absent.
2. **Step 2c closed-issue dedup search.** Proactive `bd search` + `bd list --status=closed` over cluster keywords to surface already-done work before proposing a fresh epic.
3. **Pattern H programmatic stale-claim check.** `bd show <id> --json | jq .updated_at` + age computation + threshold table (<3d healthy, 3-7d note, >7d flag, >14d release).
4. **Step 4 `apply.sh` artifact.** Generated executable script alongside the triage bead. Uses `$(bd create … --json | jq -r .id)` for epic-id capture so the reparent batch runs in one go.
