# Session Report — 2026-06-04

## Summary

Shipped `xt spec` v1 end-to-end — a PRD-level intake CLI that compiles `spec.yaml` artifacts into planner-bead input for the specialists pipeline. Six subcommands land (`draft`, `validate`, `doctor`, `apply`, `status`, `archive`) backed by a zod-as-SSOT v1 schema, a pure-logic validator with 8 gates (schema, scope-vagueness, requirement-testability, layer-hint inference, dep-cycle via Kahn, SCRUTINY floor-can-only-raise, open-question gate, plus a yaml-parse boundary), a runtime readiness probe over deployed planning + test-planning skills, an XML `<change-contract>` (D30 shape) transform, an atomic `.apply-state.json` sidecar around `sp run planner --background`, a comment-preserving yaml-AST writer for reconcile, a 7-gate archive (epic-closed + child-closed + review-evidence kv for high/critical + drift-check + immutable snapshot) and a guard test that fails the suite if `sp chain approve` or `bd update --claim` ever leaks into the spec code paths. The architectural commitment: `xt spec` is a *transform-and-dispatch* layer — it produces **one** planner bead per spec.yaml and dispatches the planner specialist; the planner itself does the bd swarm/molecule/test-issue decomposition through planning + test-planning. The composition gate (`sp chain review/approve`) stays the operator's call by design. MVP2 (apply pipeline) is runtime-gated on the 8-capability readiness probe; today 6 of 8 capabilities are missing in deployed skills (D23/D26/D28/D30 work owned by `~/dev/specialists`), so `apply` correctly refuses with exit 65 — a feature, not a bug. ## Problems Encountered

| Problem | Root Cause | Resolution |
|---------|-----------|-----------|
| `import.meta.dirname` undefined in built CJS bundle | tsup CJS target doesn't populate `import.meta.dirname` at runtime | Stopped resolving schema path from runtime; emit the stable `$id` URL (`https://xtrm.dev/schemas/spec-v1.json`) into drafted yaml; schema-store cache handles editor lookup. |
| ajv constructor not callable under NodeNext | `ajv` ships CJS; under `"module": "NodeNext"` TS resolves the default to a namespace, not a constructor | Local `AjvCtor` cast through `unknown` recovers the constructor type; pattern documented inline. Same trick will repeat for any CJS-only validator dep. |
| Template TODOs parsed as nested mappings by yaml | `TODO:` followed by space inside a list item became `- {TODO: <rest>}` | Replaced all template colons with em-dashes (`TODO — …`). Cheaper than quoting every value. |
| Schema drift between `cli/src/spec/templates/` yaml mirrors and inline strings | Two sources of truth | Deleted the on-disk yamls; inline `cli/src/spec/templates.ts` is now SSOT (matches schema.ts pattern). |
| Readiness probe in mock-repo test refused because `findRepoRoot` resolved to xtrm-tools, not tmpdir | `findRepoRoot` walks up looking for `skills/` + `hooks/` or `.xtrm/registry.json` — neither present in a fresh tmpdir | Added `XT_SPEC_REPO_ROOT` env override (consumed by both `doctor` and `apply`) to inject the probe root in tests. |
| Composition-gate guard test caught literal "sp chain approve" in `handoff.ts` doc comment | Guard regex didn't exclude comments | Reworded the doc comment to not contain the literal phrase. Acceptable bound — the guard's whole point is "this phrase shouldn't appear anywhere." |
| 3 pre-existing pi-runtime test failures in `pi-runtime-safeguards` / `pi-runtime` | Unrelated to spec work; depend on global pi state | Left as-is and noted explicitly. |

### Architectural decisions embedded in the diff
- **zod as schema SSOT**, JSON Schema artifact committed + drift-test enforced. Reusable pattern for any future spec/contract surface.
- **Inline templates over on-disk yamls** to keep tsup bundle self-contained and avoid two-source-of-truth drift.
- **Env hooks for test injection** (`XT_SPEC_SP_BINARY`, `XT_SPEC_BD_BINARY`, `XT_SPEC_REPO_ROOT`) instead of a DI framework — simple, observable, low blast radius.
- **Comment-preserving yaml writes** via `parseDocument` (eemeli/yaml AST) for reconcile and archive — `parse`+`stringify` would clobber operator comments.
- **Code-grep guard test** as the structural enforcement of the composition gate — cheaper than runtime assertions and impossible to refactor away accidentally.
- **`--dry-run` skips readiness probe** because dry-run is for inspecting XML, not dispatch — keeps the inspect-path usable while the upstream is unaligned.
